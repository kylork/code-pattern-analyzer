Factory Method Pattern

The Factory Method pattern is a creational design pattern that provides an interface for creating objects in a superclass, but allows subclasses to alter the type of objects that will be created. It defines a method for creating objects, but lets subclasses decide which class to instantiate.

Pattern Components

  • Product - Defines the interface of objects the factory method creates.
  • ConcreteProduct - Implements the Product interface.
  • Creator - Declares the factory method, which returns an object of type Product. Creator may also define a default implementation of the factory method that returns a default ConcreteProduct object.
  • ConcreteCreator - Overrides the factory method to return an instance of a ConcreteProduct.

Key Benefits

  • Loose Coupling - The code that uses the factory method doesn't need to know which concrete classes it works with.
  • Single Responsibility Principle - You can move the product creation code into one place in the program, making the code easier to support.
  • Open/Closed Principle - You can introduce new types of products into the program without breaking existing client code.
  • Extensibility - Makes the framework more customizable and allows for future expansion.

Common Applications

  • Creating UI elements that need to differ based on the operating system
  • Framework-level object creation where clients need to extend capabilities
  • Resource management systems (file, network, database connectors)
  • Plugin architectures
  • Parameterized object creation based on configuration or environment

Implementation Details

Core Components

Creator Interface/Abstract Class
Defines the factory method that returns objects of the Product type. Often includes:
  • factoryMethod() - abstract or with default implementation
  • Template methods that use the factoryMethod()
ConcreteCreator Implementation
Overrides the factory method to create specific product objects:
  • factoryMethod() - returns a specific ConcreteProduct
  • May contain additional business logic
Product Interface
Defines the interface that all products must implement:
  • Common operations that concrete products will implement
  • Ensures all products can be used interchangeably
ConcreteProduct Implementation
Implements the Product interface:
  • Specific implementation of the required operations
  • Product-specific behaviors and attributes

from abc import ABC, abstractmethod

# Product interface
class Document(ABC):
    @abstractmethod
    def open(self):
        pass
    
    @abstractmethod
    def save(self):
        pass

# Concrete products
class PDFDocument(Document):
    def open(self):
        return "Opening PDF document"
    
    def save(self):
        return "Saving PDF document"

class WordDocument(Document):
    def open(self):
        return "Opening Word document"
    
    def save(self):
        return "Saving Word document"

class TextDocument(Document):
    def open(self):
        return "Opening text document"
    
    def save(self):
        return "Saving text document"

# Creator (Factory) class
class DocumentFactory:
    @staticmethod
    def create_document(doc_type):
        """
        Factory method to create documents based on type.
        """
        if doc_type == "pdf":
            return PDFDocument()
        elif doc_type == "word":
            return WordDocument()
        elif doc_type == "text":
            return TextDocument()
        else:
            raise ValueError(f"Unknown document type: {doc_type}")

# Usage example
if __name__ == "__main__":
    # Create documents using the factory
    pdf = DocumentFactory.create_document("pdf")
    word = DocumentFactory.create_document("word")
    text = DocumentFactory.create_document("text")
    
    # Use the documents
    print(pdf.open())
    print(word.save())
    print(text.open())

// Alternative: Simple Factory implementation
class DocumentFactory {
    static createDocument(type: string): Document {
        switch(type) {
            case 'pdf':
                return new PDFDocument();
            case 'word':
                return new WordDocument();
            case 'text':
                return new TextDocument();
            default:
                throw new Error(`Unknown document type: ${type}`);
        }
    }
}

// Usage
function clientCode() {
    // Using the Factory Method pattern
    const creators: DocumentCreator[] = [
        new PDFDocumentCreator(),
        new WordDocumentCreator(),
        new TextDocumentCreator()
    ];
    
    for (const creator of creators) {
        const document = creator.createDocument();
        console.log(document.open());
        console.log(document.save());
    }
    
    // Using the Simple Factory
    const pdf = DocumentFactory.createDocument('pdf');
    const word = DocumentFactory.createDocument('word');
    const text = DocumentFactory.createDocument('text');
    
    console.log(pdf.open());
    console.log(word.save());
    console.log(text.open());
}

Implementation Considerations

  • Default vs. Parameterized Factory Methods: Factory methods can create objects based on parameters or return default objects.
  • Creator Hierarchies: Consider when to create parallel hierarchies of creators and products.
  • Template Methods: Factory methods are often called from template methods in the creator class.
  • Language Specifics: In some languages, the factory method might be a static method (Simple Factory variant).
  • Naming Conventions: Use clear and consistent naming for factory methods ("create", "make", "build", etc.).

Variations

Simple Factory
A non-pattern variation where a static method creates objects without subclassing the creator.
Parameterized Factory
Factory method that takes parameters to determine which product to create, rather than having separate creator subclasses.
Abstract Factory
Related pattern that provides an interface for creating families of related objects without specifying their concrete classes.
Factory Function
In functional programming, a function that creates and returns objects without using classes or inheritance.

Real-World Examples

UI Frameworks

UI frameworks often use Factory Method to create UI components appropriate for different platforms while presenting a consistent interface.

// UI Framework example
abstract class UIFactory {
    public abstract Button createButton();
    public abstract Checkbox createCheckbox();
}

class WindowsUIFactory extends UIFactory {
    @Override
    public Button createButton() {
        return new WindowsButton();
    }
    
    @Override
    public Checkbox createCheckbox() {
        return new WindowsCheckbox();
    }
}

class MacUIFactory extends UIFactory {
    @Override
    public Button createButton() {
        return new MacButton();
    }
    
    @Override
    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}

// Client code
public void createUI(UIFactory factory) {
    Button button = factory.createButton();
    Checkbox checkbox = factory.createCheckbox();
    button.render();
    checkbox.render();
}

// Usage
UIFactory factory;
if (System.getProperty("os.name").contains("Windows")) {
    factory = new WindowsUIFactory();
} else {
    factory = new MacUIFactory();
}

createUI(factory);

Database Connectors

Object-Relational Mapping (ORM) frameworks use Factory Method to create appropriate database connections and query builders.

// Database connector example
abstract class DatabaseConnector {
    public abstract Connection createConnection(String connectionString);
    public abstract QueryBuilder createQueryBuilder();
    
    public executeQuery(String query) {
        Connection connection = createConnection("default:connection:string");
        QueryBuilder builder = createQueryBuilder();
        String formattedQuery = builder.format(query);
        return connection.execute(formattedQuery);
    }
}

class MySQLConnector extends DatabaseConnector {
    @Override
    public Connection createConnection(String connectionString) {
        return new MySQLConnection(connectionString);
    }
    
    @Override
    public QueryBuilder createQueryBuilder() {
        return new MySQLQueryBuilder();
    }
}

class PostgreSQLConnector extends DatabaseConnector {
    @Override
    public Connection createConnection(String connectionString) {
        return new PostgreSQLConnection(connectionString);
    }
    
    @Override
    public QueryBuilder createQueryBuilder() {
        return new PostgreSQLQueryBuilder();
    }
}

// Usage
DatabaseConnector connector = Configuration.getDatabaseType().equals("MySQL") 
    ? new MySQLConnector() 
    : new PostgreSQLConnector();

connector.executeQuery("SELECT * FROM users");

Logistic Management

Logistics applications use Factory Method to create different types of transport without coupling the client code to specific transport classes.

// Logistics application example
interface Transport {
    void deliver();
}

class Truck implements Transport {
    @Override
    public void deliver() {
        System.out.println("Delivering by land in a truck");
    }
}

class Ship implements Transport {
    @Override
    public void deliver() {
        System.out.println("Delivering by sea in a ship");
    }
}

abstract class LogisticsCompany {
    // Factory method
    protected abstract Transport createTransport();
    
    public void planDelivery() {
        Transport transport = createTransport();
        System.out.println("Planning delivery route...");
        transport.deliver();
    }
}

class RoadLogistics extends LogisticsCompany {
    @Override
    protected Transport createTransport() {
        return new Truck();
    }
}

class SeaLogistics extends LogisticsCompany {
    @Override
    protected Transport createTransport() {
        return new Ship();
    }
}

// Usage
class LogisticsApp {
    public static void main(String[] args) {
        LogisticsCompany logistics;
        
        // Determine logistics type based on delivery location
        if (DeliveryLocationService.isOverseas()) {
            logistics = new SeaLogistics();
        } else {
            logistics = new RoadLogistics();
        }
        
        logistics.planDelivery();
    }
}
